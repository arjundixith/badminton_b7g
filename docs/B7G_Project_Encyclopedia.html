1. Project Overview

This product is a tournament operations platform for badminton league matches. It provides referee workflows, live viewer dashboards, standings, finals orchestration, and post-finals category analytics.

Tournament Format

10 round-robin ties (league stage). Each tie has 12 regular games. Game #13 appears only as a decider at 6-6 tie score.

Qualification

After all 10 ties are complete, rank 1 and 2 become finalists. Rank 3 automatically gets Bronze.

Finals

Final tie consists of 12 games between top 2 teams. Winner gets Gold, other finalist gets Silver.

Live Data Model

Viewer and referee screens poll backend frequently; updates propagate without manual refresh.

2. Prerequisites and Local Installation

2.1 Tools Required

  • Python 3.11+ (backend)
  • Node.js 18+ and npm (frontend)
  • Git
  • PostgreSQL 14+ (required for local runtime)
  • pgAdmin 4 or DBeaver (recommended local DB GUI)

2.2 Clone and Install

git clone <your-repo-url>
cd badminton-app

# Local Postgres (Homebrew example)
brew services start postgresql@16
createdb -h localhost -U <your-macos-username> badminton_b7g || true

# Backend
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
export DATABASE_URL=postgresql://<your-macos-username>@localhost:5432/badminton_b7g
python3 seed.py
uvicorn app.main:app --reload

# Frontend (new terminal)
cd ../frontend
npm install
cp .env.example .env
# set VITE_API_URL=http://localhost:8000
npm run dev
Result: frontend runs on http://localhost:5173, backend on http://localhost:8000.

2.3 pgAdmin Setup (Local DB UI)

  1. Install pgAdmin: brew install --cask pgadmin4
  2. Create server with Name Badminton Local PG (any label).
  3. Use Host localhost, Port 5432, Maintenance DB postgres.
  4. Username: macOS user for Homebrew Postgres (for example arjundixithts) or postgres for Docker Postgres.
  5. Password: your configured local role password (Docker default is postgres).

2.4 Seed Modes

  • python3 seed.py: fresh tournament start (same teams/players, all pending).
  • python3 seed.py --demo-progress: sample live/completed demo states.

3. How the System Works End-to-End

3.1 Runtime Request Flow

  1. User action in React page (for example, Save Score in Referee).
  2. frontend/src/api.js sends HTTP request.
  3. FastAPI route in backend/app/routes/*.py receives request.
  4. Route calls business logic in backend/app/crud.py.
  5. CRUD updates DB and recalculates tie/final status as needed.
  6. Viewer/Referee polling reloads dashboard and reflects latest state.

3.2 Core Rule Enforcement

  • Referee assignment required before scoring.
  • Lineup confirmation required before scoring.
  • Badminton scoring: 21 with deuce, cap 30.
  • Game #13 only appears/starts when tie is 6-6.
  • Final tie appears only after league completion.

6. Deployment Guide

6.1 Single Free Deployment on Vercel (Current Setup)

  1. Import repo into Vercel with root directory as repository root.
  2. Vercel uses vercel.json to build frontend and route API.
  3. Set env vars: VITE_API_URL=/api, DATABASE_URL, CORS_ORIGINS, AUTO_SEED_ON_EMPTY=true.
  4. Deploy and verify /api/health and /api/viewer/dashboard.

6.2 Why /api Rewrite Exists

The project rewrites /api/(.*) to /api?__path=/$1 and then maps __path inside api/index.py. This avoids Vercel nested-path edge NOT_FOUND behavior in some serverless route layouts.

6.3 Optional Split Deployment

  • Frontend on Vercel.
  • Backend on Render using render.yaml.
  • Set frontend VITE_API_URL to Render URL.

7. Operations Runbook

7.1 One-time Full Data Reset

Use carefully: Set AUTO_SEED_FORCE_RESET=true for one deploy/startup to reset tournament state to default seed. Then set it back to false.

7.2 Verification Checklist

  • /api/health returns JSON status ok.
  • Viewer loads without JSON parse errors.
  • Referee can assign and score a match end-to-end.
  • Standings update after completed tie games.
  • Final appears only after all 10 ties completed.
